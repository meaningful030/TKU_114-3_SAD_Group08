## 如何開啟與操作

1. 直接開啟 [第二個可操作切片](revised_or_second_working_slice/index.html)。
2. 在攤主頁填入攤位名稱、作品類型與有效 email，送出申請。
3. 確認申請編號、送件時間與審核中狀態。
4. 切換「主辦方」，在審核區選擇通過；或先填審核意見再退回。
5. 切回「攤主」，確認最新狀態與意見同步；若為退回，可修改原申請並以同一編號重新送審。

重新整理會清除資料；本原型不代表正式送件、保存、登入或權限控管。

## GitHub 提交紀錄

附件為 GitHub 上傳封裝檔，不包含 `.git` 提交歷程。請解壓縮後，在小組儲存庫中建立同名 `0714_User_Stories_Acceptance_Implementation/` 資料夾，並依下列順序分三批上傳，避免所有成果只出現在同一筆提交：

1. 文件批次：上傳需求與規格 Markdown 文件，提交訊息使用 `docs: 完成使用者故事與需求文件包初稿`。
2. 功能批次：上傳 `revised_or_second_working_slice/`，提交訊息使用 `feat: 完成第二個可操作切片`。
3. 測試批次：上傳 `manual_acceptance_test.md`、`evidence/` 及最後更新的紀錄文件，提交訊息使用 `test: 補上手動驗收測試與修正紀錄`。

建議提交訊息：

```text
docs: 完成使用者故事與需求文件包初稿
feat: 完成第二個可操作切片
test: 補上手動驗收測試與修正紀錄
```

| 提交分類 | 實際 commit 識別碼 |
|---|---|
| docs | **【需你完成第 1 次提交後填寫】** |
| feat | **【需你完成第 2 次提交後填寫】** |

> commit 識別碼只有 GitHub 接受提交後才會產生；本封裝檔不預填假編號。若課程要求 README 顯示三筆實際 ID，第三次提交後需再編輯本表並補一筆小型文件更新提交。
