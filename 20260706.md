## 任務 1：案例拆解 — 社團活動報名與場地申請系統

```
小組名稱：第8組

利害關係人（Stakeholders）：
1.社長和幹部:辦活動的人,負責填報名表和申請場地
2.一般社員:報名參加活動,查活動資訊和自己有沒有報到
3.課外活動指導組承辦人:審核活動申請,管全校社團的事務
4.場地管理單位(總務處或各館的管理員):管場地借用,維護哪些時段能借
5.指導老師:活動申請要經過他審閱簽核
6.系統管理員:管帳號和權限,維護系統

痛點（Pain Points）：
1.已核准的場地時段散在紙本和各承辦人手上,沒辦法馬上查,常常重複核准,活動當天撞場
2.報名用紙本或即時通訊收,名單很散,超額也沒有候補機制,幹部要自己人工整理,容易漏登或重複
3.紙本申請和紀錄不好保存也不好找,社團一年一交接,歷史資料常常整批不見
4.送出申請之後不知道審到哪一關了,只能一直打電話問承辦人

資料（Data）：
1.活動資料(名稱、時間、地點、人數上限、負責人、活動說明)
2.報名紀錄(誰報名了哪個活動、什麼時候報的、報名狀態)
3.候補名單(超額報名時的遞補順序)
4.場地資料和時段狀態(可借時段、已核准時段、容納人數)
5.場地申請紀錄(哪個社團、哪個時段、用途)
6.審核紀錄(申請單狀態、審核人、時間和意見)
7.使用者帳號和角色權限

業務規則（Business Rules）：
1.活動和場地要在活動開始前至少3個工作天提出申請
2.同一個場地同一個時段只能核准一件申請,送審前系統要先擋掉時段衝突的
3.要照「指導老師→課外組→(場地單位)」的順序審核通過,活動才能公告和開放報名
4.報名滿了之後自動進候補,有人取消就照順序遞補並通知
5.學生個資只有活動負責人和審核人員能看,不能公開或匯出

風險（Risks）：
1.大家已經習慣紙本和私訊,如果沒有強制規定或誘因,系統上線可能沒人用,紙本和線上兩邊跑反而更亂
2.學期初大家一起送申請,如果審核人力或系統效能跟不上會塞件,搞不好比紙本還慢
3.場地狀態如果沒即時更新(像學校臨時徵用場地),系統寫的跟現場不一樣,大家就不會再信這個系統
```

## 任務 2：候選題目發想

```
候選題目 1
系統名稱：投資策略回測與自動交易系統
目標使用者：想驗證自己交易規則,但不會寫程式的散戶
要解決的問題：很多人買賣股票是憑感覺,沒辦法知道自己的規則放到歷史資料上到底賺不賺,也很難有紀律地執行
核心任務：
1.用表單設定進出場條件,不用寫程式
2.用歷史資料跑回測,輸出勝率、期望值這些績效數字
3.接券商API,把訊號變成自動下單
本期不處理：預測股市走向、投資建議、保證獲利
```

```
候選題目 2
系統名稱：旅遊預算追蹤系統
目標使用者：自助旅行的人和同行旅伴
要解決的問題：出去玩大家一起花錢,很難即時知道總共花多少,常常不知不覺就超支,回來分帳又算不清
核心任務：
1.建立旅程、總預算和分類預算(交通/住宿/餐飲/購物)
2.記帳,即時顯示剩多少,快超支就警示
3.多人分帳,算出每個人該付或該收多少
本期不處理：實際付款金流、匯率自動更新(匯率手動輸入就好)
```

```
候選題目 3
系統名稱：同人誌販售會攤位管理系統
目標使用者：販售會主辦方、攤主、參加者
要解決的問題：攤位申請都用email和表單分散在收,主辦方用試算表人工排位容易出錯;
公告出去的攤位資訊常常跟現場不一樣,參加者在場館裡迷路,找不到想去的攤
核心任務：
1.攤主線上交攤位申請,可以查審核進度
2.主辦方審核,手動排攤位位置,系統幫忙檢查有沒有排到同一格
3.參加者用場館地圖查攤位的正確位置
本期不處理：R18分區審查、線上金流、攤位商品內容管理
```

### D. 目標使用者（Target Users）

| 角色 | 他/她在系統中的任務 | 目前痛點 |
|---|---|---|
| 主辦方 | 收攤位申請、審核、排攤位位置、發布場館地圖 | 申請散在email和表單裡,人工整理容易漏件;排位用試算表,很難發現排重複或衝突 |
| 攤主 | 線上交攤位申請、查審核結果和自己攤位的位置 | 申請交出去之後不知道審到哪;活動前看不到自己的位置,不好規劃佈置和宣傳 |
| 參加者 | 用場館地圖查攤位位置和活動資訊、活動後填回饋 | 場刊和網路上的資訊常跟現場不一樣,容易迷路、排錯隊、錯過想買的攤 |

次要角色說明:系統管理員(管帳號權限)和現場工作人員算次要的利害關係人,
這期原型不幫他們做獨立介面,只在帳號權限設計時先留角色欄位。
場館是外部單位,這個系統不管場地本身,只用主辦方上傳的平面圖,所以不列進使用者。

### E. 問題陳述（Problem Statement，任務 4）

```
對於同人誌販售會的參加者來說,現在攤位資訊主要靠紙本場刊和社群貼文公告,
常常發生官方更新不及、攤位編號跟現場實際位置對不起來的狀況,
結果參加者在場館裡迷路、排錯隊、錯過限量商品;攤主因此流失客人,
主辦方也要花很多人力在現場回答「某某攤在哪」。
會變成這樣的上游原因,是主辦方用email和試算表人工管理申請和排位,容易出錯又很難同步。
本專案想設計一個同人誌販售會攤位管理系統,讓主辦方把申請、審核、排位集中在一個地方管理,
參加者查到的位置就是現場實際的位置,希望能縮短找攤時間、減少現場詢問量。
本期聚焦在「攤位申請、審核、配置、地圖查詢」這段流程,金流和商品管理都不碰。
```

### F. 專案目標（Project Goal）

```
做出一個攤主能線上申請、主辦方能集中審核和排位、參加者查得到正確攤位位置的系統原型,
期末用模擬資料把「申請→審核→排位→地圖查詢」整條流程跑給大家看,排位結果不能有任何重複。
```

### G. 範圍內項目（In-Scope，含優先順序）

P1是期末展示一定要有的,P2是有時間再做,時間不夠就先砍P2。

1.【P1】攤主線上提交攤位申請表單,並可查詢審核進度
2.【P1】主辦方審核申請(通過/退回,附審核意見)
3.【P1】主辦方手動排攤位位置,系統自動檢查有沒有排到同一格
4.【P1】場館地圖與攤位查詢(可以用攤位名稱、編號或作品類型搜尋)
5.【P2】活動基本資訊頁(時間、地點、參加方式)
6.【P2】參加者報名
7.【P2】參加者活動後回饋表單

### H. 範圍外項目（Out-of-Scope）

1.線上付款與金流(攤位費、門票)
2.攤位商品品項與庫存管理
3.R18分區與內容審查
4.票務驗票與入場管制
5.活動經費規劃、工作人員招募排班
6.自動動線最佳化演算法(這期只做手動排位+衝突檢查)
7.即時通知(email或推播),這期先讓使用者自己上系統查

### I. 原型核心任務（Core Tasks for Prototype）

| 核心任務 | 主要角色 | 任務起點 | 任務終點 | 初步驗收標準 |
|---|---|---|---|---|
| 1. 攤主提交攤位申請 | 攤主 | 填寫並送出申請表單 | 申請出現在主辦方的待審清單,攤主端看得到「審核中」 | 送出後主辦方那邊馬上看得到;必填欄位沒填完就不能送出 |
| 2. 主辦方審核並排攤位 | 主辦方 | 打開待審清單 | 所有通過的申請都拿到一個不重複的位置編號 | 每件通過的申請都有位置;想把兩攤排到同一格時,系統要擋下來 |
| 3. 參加者查攤位位置 | 參加者 | 在地圖輸入攤位名稱或編號 | 地圖標出正確位置 | 隨便抽10個攤查,地圖標的位置要跟配置表完全一樣 |

補充說明:

- 衝突的定義:同一個位置編號被排給兩個以上的攤。主辦方在排位畫面存檔時,
  系統馬上檢查那一格有沒有被占,有就擋下來並提示。
- 地圖打算用「靜態平面圖+格子編號」的方式做(自己畫簡化的格狀圖就好),
  查詢結果直接在圖上疊一個標記,不做互動式的地圖編輯器,不然5週做不完。
- 攤主查審核進度算任務1的延伸(送出後就會產生狀態),不另外算一個核心任務。

### J. 初步資料物件（Initial Data Objects）

| 資料物件 | 說明 | 誰建立 | 誰修改 | 誰查看 |
|---|---|---|---|---|
| 使用者帳號 | 各角色的帳號和權限(主辦方/攤主/參加者) | 使用者本人 | 本人(基本資料) | 主辦方(只看得到必要欄位) |
| 活動基本資訊 | 場次名稱、日期、地點、參加方式 | 主辦方 | 主辦方 | 公開 |
| 攤位申請書 | 攤主填的申請資料(社團名、作品類型、需求) | 攤主 | 攤主(截止前) | 主辦方、該攤主本人 |
| 審核結果紀錄 | 每件申請通過或退回的結果和意見 | 主辦方 | 主辦方 | 主辦方、該攤主本人 |
| 攤位配置表 | 通過的攤位對應到哪個位置編號 | 主辦方 | 主辦方 | 主辦方;攤主看得到自己的位置 |
| 場館地圖 | 場館平面圖和攤位位置標示 | 主辦方 | 主辦方 | 公開 |
| 參加者報名紀錄 | 報名這次活動的參加者名單 | 參加者 | 參加者(截止前) | 主辦方 |
| 活動回饋紀錄 | 參加者活動後填的回饋 | 參加者 | 不能改 | 主辦方 |

資料之間的關聯:申請書連到攤主帳號;審核結果和配置表連到申請書;
地圖上的標記是讀配置表畫出來的;報名和回饋連到參加者帳號和活動場次。

### J-1. 補充業務規則（Business Rules）

1.申請截止後攤主不能自己改申請書,要改的話得經過主辦方同意,由主辦方代改
2.同一格同一場次只能排一個攤(系統強制擋)
3.審核通過而且排好位置的攤,才會出現在公開地圖上
4.攤位位置一律以配置表為準,公開地圖直接讀配置表最新的版本;
  臨時換位置只要改配置表,地圖就跟著變,不會有兩邊不一致的問題
5.攤主和參加者的個資只有主辦方能看,系統不提供匯出

### K. 初步成功標準（Initial Success Criteria）

1.展示當天用至少10筆模擬申請把「申請→審核→排位」整個流程跑完,排位結果沒有任何一格重複
2.隨便抽10個攤查地圖,標的位置跟配置表要完全一致
3.找5個人照著地圖找指定的攤,平均3分鐘內找到(模擬情境),或回饋問卷滿意度平均4分以上(滿分5分)
4.攤主審核完成後就能上系統看到自己的位置,不用等到活動當天
