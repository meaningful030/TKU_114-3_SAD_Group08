# 任務 2：使用案例清單

需求基準：`0.8-draft／2026-07-15`  
參與者目標來源：`task1_participant_goal_matrix.md`  
專案：第 8 組－同人誌販售會攤位管理系統「攤集」

## 1. 使用案例整理原則

- 使用案例名稱採「動詞＋名詞」，描述參與者想完成的完整目標。
- 欄位驗證、顯示清單、按下按鈕、更新計數與地圖標記是案例中的步驟、規則或結果，不獨立當成使用案例。
- 「攤集」是受分析的系統，不列為參與者；目前也沒有已確認會直接互動的外部服務。
- 支援參與者欄若為「無」，表示沒有其他人在同一次使用案例中直接與系統互動；不把後續使用資料的人誤列為支援參與者。
- 優先順序沿用已確認的 MoSCoW、D-01／D-02 與切片狀態。D-02 尚未完成其餘需求的全面排序，因此不自行編造所有案例的固定名次。

## 4. 建模與完整描述邊界

- UC-01 可描述目前既有欄位及錯誤處理，但正式欄位、攤名字數、email 規則、編號格式與持久保存標為 `【待確認：UQ-01】`。
- UC-03 只描述已確認的通過／退回與意見規則；正式審核角色、順序、判準、撤銷及完整歷程標為 `【待確認：UQ-04】`。
- UC-04 不加入補件期限、重送上限或版本歷程，標為 `【待確認：UQ-03】`。
- UC-05 目前只確認「同一格位不得重複指派並保留原配置」；其餘配置政策標為 `【待確認：UQ-05】`。BR-05 不得直接寫入基本流程。
- UC-06 依 BR-06 使用完整名稱、完整編號、完整作品類型及無結果，要求位置一致且公開結果不含 email；模糊搜尋與作者別名標為 `【待確認：UQ-02】`，正式量化目標標為 `【待確認：UQ-06】`。
- 正式登入、資料庫、通知、自動排位及互動式地圖均不得增加為使用案例或案例步驟。
