## 2. 改善理由摘要

| 改善 | 採用理由 | 沒有採用的過度延伸 |
|---|---|---|
| I1 | 直接回應攤主「是否收件、目前狀態、退回怎麼處理」三個可追蹤目標 | 不自行加入補件截止日、提醒通知、重送上限或完整版本歷程 |
| I2 | 讓同一筆有效申請直接形成待審案件，移除人工複製交接 | 不宣稱已有後端、資料庫、正式帳號或跨裝置保存 |
| I3 | 只防止已有明確來源的「同格重複」，並保留人工選位責任 | 不加入自動排位、禁排格、多格或特殊衝突演算法 |
| I4 | 先確保系統內的文字位置與地圖標記一致，具明確 BR-06／AC-07-05 依據 | 不把紙本、社群、通知及版本管理畫成已確認功能 |
| I5 | 以已確認的三種完整值搜尋支援自助查攤，並保留人工備援 | 不加入模糊搜尋、作者別名、導航或互動式地圖 |

## 3. 利害關係人確認清單

以下項目不阻礙本次對照表完成，但會影響後續需求定稿或流程擴充：

| 待確認項目 | 主要利害關係人 | 不確認時的建模處理 |
|---|---|---|
| UQ-01 正式欄位、格式、編號與保存 | 攤主、主辦方承辦人 | 只沿用目前固定驗收資料及原型規則 |
| UQ-02 模糊比對、大小寫、去空白與別名 | 一般參加者、主辦方承辦人 | 只支援 BR-06 的完整值搜尋 |
| UQ-03 補件期限、重送上限與版本歷程 | 攤主、主辦方承辦人 | 退回可重送，但不加入期限、上限或歷程承諾 |
| UQ-04 正式角色、權限、撤銷與審核順序 | 主辦方承辦人、系統維護者 | 只標示業務責任，不把角色切換視為登入 |
| UQ-05 排位前置、禁排格、多格與特殊衝突 | 主辦方承辦人、場館方 | 只驗證同格不得重複及原配置保留 |
| UQ-06 效能、一致率及測試環境 | 一般參加者、主辦方承辦人 | 不把候選數值寫成正式完成門檻 |
| 是否擴充紙本／社群／通知同步 | 主辦方承辦人、現場工作人員、攤主 | I4 維持部分改善，明列外部副本仍可能過期 |

## 4. 證據使用原則

- 需求證據只採用 Project Charter v2、Stakeholder Feasibility v1、需求來源表、使用者故事、功能需求、業務規則／資料限制、驗收條件、模擬訪談及使用者已確認的 BR-06。
- 模擬訪談屬中等強度證據；本表不把它宣稱為特定真實活動的正式 SOP。
- `updated_code_agent_brief.md`、implementation backlog、程式碼代理紀錄及代理提出的功能想法，只能協助檢查實作狀態，**沒有作為新增需求、改善理由或業務規則的證據**。
- 尚未由來源或利害關係人確認的內容，一律以 UQ 編號或「範圍外／部分改善」標示。
