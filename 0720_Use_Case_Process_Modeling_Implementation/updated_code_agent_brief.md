## 5. 輸入、輸出與狀態變化

### 5.1 輸入

| 類型 | 欄位 | 規則 |
|---|---|---|
| 申請資料 | `id`、`boothName`、`category`、`categoryOther`、`email`、`note`、`time` | 延用現有結構，不新增正式欄位 |
| 審核輸入 | `decision`、`comment` | decision 僅可使用既有「通過／退回」；退回意見 trim 後不得為空 |
| 狀態資料 | `status`、`reviewComment`、`resubmissionCount` | status 僅可使用「審核中／通過／退回」 |
| 示範資料 | `reviewedAt` | 可保留既有工作階段顯示，但不得作為排序、分流、權限或驗收判斷依據 |

### 5.2 輸出

- `pendingApplications`：只含 `status === '審核中'` 的原申請物件參照或唯讀投影。
- `reviewedApplications`：只含 `status === '通過'` 或 `status === '退回'` 的原申請物件參照或唯讀投影。
- 待審筆數必須等於 `pendingApplications.length`。
- 待審表與已審結果表必須互斥；同一狀態下同一編號不可同時出現在兩區。
- 已審結果開啟後輸出案件核心資料、狀態及完整意見；操作控制須唯讀或停用。
- 攤主狀態卡輸出同一案件狀態；退回時逐字顯示完整意見。

### 5.3 狀態轉移

```text
審核中 --通過--> 通過
審核中 --有效退回--> 退回
審核中 --無效退回--> 審核中
退回 --合法補正重送--> 審核中
```

不得新增「已審」、「草稿」、「已取消」、「待補件」等新案件狀態。「待審」與「已審」只是清單投影名稱，不是 `status` 新值。

## 6. 保留、修正、新增與刪除項目

### 6.1 保留

- 保留單一 `applications` 陣列作為工作階段資料來源。
- 保留 `validateDraft`、`applyReview`、`countPending`、`toOwnerStatus`、`displayCategory`、`reviseApplication` 的既有對外行為。

### 6.2 修正

- 修正 `renderApplications()`：待審區只渲染待審投影，不再渲染全部 `applications`。
- 修正待審空狀態：以待審投影長度判斷，不以全部案件數判斷。
- 修正審核完成後的重新渲染，使通過／退回案件立即離開待審區。
- 修正補正重送後的重新渲染，使同一案件由已審投影移回待審投影。
- 必要時修正標題與按鈕文字，使「開始審核」只出現在待審區，「查看結果」只出現在已審區。

### 6.3 新增

- 在 `domain.js` 新增明確的純查詢函式，建議名稱：
  - `getPendingApplications(applications)`
  - `getReviewedApplications(applications)`
- 在主辦方畫面新增與待審區明確分開的「已審結果」區及空狀態。
- 已審列新增「查看結果」入口；開啟後內容唯讀，通過／退回按鈕不可操作。
- 在 `domain.test.js` 新增待審／已審投影與三條狀態路徑的回歸測試。
- 新增本次人工驗收紀錄及可辨識的待審／已審畫面證據。

函式名稱若因既有命名慣例需要不同名稱，可以等義命名，但輸入、輸出及純函式行為不得改變。

### 6.4 刪除或移除

- 從「待審」畫面結果中移除通過／退回案件列；資料本身不得刪除。
- 不得刪除既有通過／退回回查能力、申請資料或退回意見。

### 固定驗收資料

- 申請編號：`APP-001`（測試可使用現有編號產生器，但斷言須證明同一編號）。
- 有效退回意見：`請補充攤位展示說明`。
- 無效退回意見：`''` 及只含空白的字串。
- 初始狀態：`審核中`；初始待審筆數：`1`；初始重送次數：`0`。

### 最低自動測試群組

1. 混合狀態投影：待審一筆、已審兩筆、集合互斥、原資料不變。
2. 通過：審核中 → 通過；待審一筆 → 零筆；已審零筆 → 一筆。
3. 有效退回：審核中 → 退回；完整意見保留；待審零筆；已審一筆。
4. 無效退回：狀態不變；待審一筆；已審零筆；回傳既有錯誤。
5. 補正重送：退回 → 審核中；同一編號；次數加一；意見清空；投影由已審移回待審。
