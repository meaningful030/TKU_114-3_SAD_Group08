# 任務 5：現況流程與痛點分析

需求基準：`0.8-draft／2026-07-15`  
選定流程：**人工收件 → 人工審核與試算表排位 → 紙本／社群公告 → 現場查攤**

## 1. 選擇理由與現況邊界

這條流程直接對應專案章程的核心問題：上游以 email、Google 表單與試算表人工管理申請和排位，資訊再以紙本場刊、社群公告及配置表傳遞到現場，因而出現漏件、重複格位、版本不同步及參加者找不到攤位。

本圖只呈現需求文件描述的現況工具與人工做法：

- 攤主閱讀簡章，以 Google 表單或 email 送件，再等待 email／社群結果。
- 主辦方人工彙整至試算表、檢查資料、審核、排位、匯出配置表及發布公告。
- 參加者查閱紙本場刊或社群公告；資訊不符時詢問現場工作人員。
- 現場工作人員依主辦方提供的配置表口頭指引或再向主辦方查證。

沒有畫入尚未存在的集中待審系統、固定狀態查詢頁、自動欄位驗證、自動衝突警告、靜態地圖搜尋或正式登入。

## 2. 泳道與角色

| 泳道 | 現況責任 |
|---|---|
| 攤主 | 準備資料、由既有管道送件、等待通知、依通知補資料、查看位置公告 |
| 主辦方承辦人 | 接收與彙整申請、人工審核、試算表排位、匯出配置、發布或修改公告 |
| 一般參加者 | 依紙本／社群資訊找攤，資訊不符時尋求現場協助 |
| 現場工作人員 | 查閱配置表、口頭指引，版本不足時回問主辦方或現場查證 |

## 4. 角色交接

| 交接 | 從 → 到 | 交付內容 | 風險 |
|---|---|---|---|
| H1 | 攤主 → 主辦方承辦人 | Google 表單或 email 申請資料 | 多管道需人工彙整，可能漏件或重複處理 |
| H2 | 主辦方承辦人 → 攤主 | 補件通知或審核結果 | 無固定查詢入口，攤主只能等待 email／社群 |
| H3 | 主辦方承辦人 → 攤主／一般參加者／現場工作人員 | 紙本場刊、社群公告、配置表 | 多份輸出在臨時異動後可能版本不同 |
| H4 | 一般參加者 → 現場工作人員 | 口頭詢問「某攤在哪」 | 增加現場人力負擔；若工作人員版本過期仍可能答錯 |

## 5. 痛點、來源、位置與影響

| 痛點 | 流程位置 | 來源 | 影響 |
|---|---|---|---|
| P1 等待不透明 | C3、D3、C4；H2 | INT-OWN-01：送出後沒有明確收件結果、擔心漏件；INT-OWN-02：目前多半等 email 或社群公告；Project Charter v2 目標使用者痛點 | 攤主無法即時確認收件或進度，只能等待或另行確認，增加不安與往返溝通 |
| P2 多管道人工彙整 | O1、O2；H1 | Project Charter v2 問題陳述：email 與試算表人工管理容易出錯且難同步；Stakeholder Feasibility v1：email 和表單收件容易漏；INT-ORG-05：最想先改善收件到待審，因漏件影響最大 | 申請可能漏登、重複處理或在表單與試算表間資料不一致，延誤審核 |
| P3 試算表人工排位 | O6、D5、O7 | Stakeholder Feasibility v1：試算表排位怕排重複；INT-ORG-02：同一格不能給兩攤；Project Charter v2 核心問題 | 重複格位可能直到人工複核或現場才被發現，造成重排與配置爭議 |
| P4 多份輸出與臨時異動不同步 | O8～O10、D6；H3 | INT-STAFF-01：臨時更動後手上的表不同步；INT-ORG-05：臨時變更仍靠人工；Project Charter v2：紙本／社群與現場位置可能不同 | 公告、配置表與現場各自維護，修改一份可能漏改其他版本，讓錯誤資訊持續傳播 |
| P5 現場找攤與詢問負擔 | A1、D7、A3、S1～S3；H4 | Project Charter v2 問題陳述：參加者迷路、排錯隊、錯過商品，主辦方與工作人員花人力回答；Stakeholder Feasibility v1：配置表若不一致會整天被問且答錯 | 參加者浪費時間或錯過攤位；攤主流失客人；現場工作人員與主辦方承擔大量重複詢問 |

## 6. 來源強度與限制

- Project Charter v2 與 Stakeholder Feasibility v1 是小組既有文件，作為現況問題與工具的主要依據。
- `INT-OWN`、`INT-ORG`、`INT-VIS`、`INT-STAFF` 是課堂模擬訪談，來源強度為中；本圖是可供分析的現況基線，不宣稱為特定真實活動主辦方的正式 SOP。
- 「補資料後由原管道重送」只描述人工管道往返，不加入已設計的新系統狀態、補件期限或版本歷程。
