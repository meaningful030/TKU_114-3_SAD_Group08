# 任務 6：改善後目標流程

需求基準：`0.8-draft／2026-07-15`  
比較基準：任務 5「人工收件、審核排位與現場查攤」  
流程範圍：攤主準備申請 → 系統收件與主辦方人工審核 → 主辦方人工配置 → 公開靜態查攤 → 找到攤位或人工查證仍失敗。

## 1. 目標流程邊界

- 起點與現況一致：攤主準備並送出攤位申請。
- 成功終點與現況一致：一般參加者找到攤位。
- 失敗終點與現況一致：現場工作人員及主辦方仍無法及時確認位置。
- 另保留一個合法替代終點：退回後暫不補正；申請維持「退回」，日後可再觸發 UC-04。
- 本圖描述本期 P1 整體目標能力；FR-06 是後續切片、FR-07 是下一優先切片，不表示它們已在第二個可操作切片完成。
- 不加入正式登入、後端、資料庫、自動排位、互動地圖、自動通知、金流、候補或活動後功能。

## 2. 泳道與責任

| 泳道 | 目標流程責任 | 明確不承擔的責任 |
|---|---|---|
| 攤主 | 填寫與修正資料、查看收件與審核結果、決定是否補正、查看位置公告 | 不自行決定審核結果或配置格位 |
| 系統 | 驗證輸入、建立編號與狀態、同步待審、保存審核結果、拒絕重複格位、提供一致的公開結果與靜態標記 | 不自動審核、不自動排位、不替主辦方處理現場例外 |
| 主辦方承辦人 | 人工判斷通過／退回、填寫必要意見、人工選擇與調整格位 | 不把演算法當成審核或排位決策者 |
| 一般參加者 | 以完整名稱、完整編號或完整作品類型搜尋，依靜態標記前往；必要時詢問 | 不存取聯絡 email 或內部申請資料 |
| 現場工作人員 | 使用同一公開配置協助口頭指引；無法確認時回問主辦方或現場查證 | 不具專屬新增、審核或配置功能 |

權限說明：審核與配置責任只配置給「主辦方承辦人」。這是業務責任邊界，不宣稱第二切片已具正式登入或技術性權限控管；正式角色、審核權限與撤銷仍屬 `UQ-04`，並受 `CON-01` 限制。

## 6. 保留的人工判斷、權限與限制

- 審核：T-O1、T-D2a 由主辦方承辦人判斷，系統僅驗證退回意見與保存結果；不建立自動審核。
- 配置：T-O4、T-O5、T-D5、T-O6 由主辦方承辦人選位與決定臨時調整；系統只做已確認的同格衝突防護；不建立自動排位。
- 現場：T-F2～T-F4 保留口頭指引及現場查證，因靜態地圖無法涵蓋所有臨時環境問題。
- 權限：模型把審核／配置責任限於主辦方泳道；正式登入、角色驗證、撤銷及審核順序仍是 UQ-04，不把原型角色切換畫成正式登入。
- BR-05：圖中的「通過後接配置」是同一示範案件的流程銜接，不宣稱系統已強制禁止審核中／退回案件配置；是否成為強制規則仍待 UQ-05。
